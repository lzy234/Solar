# 光伏电站项目方案总览（第一版）

## 1. 项目背景
客户在上海多地投资了太阳能光伏电站，但不同电站使用的运维平台不一致，导致信息查看分散、告警管理不统一。

## 2. 项目目标（现阶段）
产出一版“看起来靠谱、可对外沟通”的方案文档（并可直接转化为 PPT 结构），用于客户经理与客户进一步确认方向。

## 3. 需求范围（按 README 对齐）
1. 平台整合：整合多个光伏电站运维平台。客户可提供 API，我方重点在后端适配不同 API，并在统一前端展示。
2. 自动化流程：接入 `OpenClaw` 作为客户秘书能力，支持电站运营周报生成、告警推送到客户通讯软件等功能。
3. 异常检测：关注脏污遮挡等导致发电量与收益下降的场景；当前尚未形成最终检测方案，方向是“数据分析 + 简单硬件加装”。

## 4. 总体方案思路
- 先统一数据与告警入口，解决“看得见、看得全”的问题。
- 再建设自动化秘书流程，解决“信息传递与运营动作自动化”问题。
- 最后逐步补齐异常检测能力，解决“收益异常早发现、可解释、可追踪”问题。

## 5. 分阶段规划
### 阶段 A：统一看板与告警汇聚（优先）
- 打通多平台 API，建立统一数据模型。
- 完成统一看板（电站状态、发电量、告警列表）。
- 形成告警中心，支持统一查询与状态跟踪。

### 阶段 B：OpenClaw 自动化流程
- 配置告警推送规则（级别、时段、重复抑制）。
- 对接 `OpenClaw`，实现“告警 -> 通讯软件推送”。
- 自动生成运营周报（日报/周报模板化输出）。

### 阶段 C：异常检测（方案迭代）
- 先做数据侧检测：发电偏差、同区域对标、趋势退化识别。
- 再按投入产出比引入轻量硬件：辐照度、温度、摄像头、灰尘相关采集。
- 逐步形成“异常识别 -> 证据输出 -> 处置建议”的闭环。

## 6. 关键能力设计
### 6.1 API 适配层
- 多厂商 API 鉴权封装。
- 字段映射与状态标准化。
- 调用失败重试、限流与日志追踪。

### 6.2 统一数据模型（建议）
- 电站：`station_id, station_name, location, capacity_kw`
- 发电：`station_id, ts, power_kw, energy_kwh`
- 告警：`alarm_id, station_id, level, code, message, ts, status`
- 异常：`anomaly_id, station_id, type, score, evidence, ts`

### 6.3 OpenClaw 编排能力
- 输入：统一告警事件、发电统计数据。
- 处理：规则判断、消息编排、周报生成。
- 输出：通讯软件告警消息、结构化运营周报。

## 7. 验收口径（第一版建议）
- 至少接入 2 个不同平台并完成统一展示。
- 告警推送链路可验证：平台 -> 规则 -> `OpenClaw` -> 通讯软件。
- 可输出基础运营周报模板（含核心指标与异常摘要）。
- 异常检测给出首批可运行规则与样例结果。

## 8. 前置条件与风险
### 前置条件
- 客户提供各平台 API 文档、鉴权信息、调用配额。
- 客户明确通讯软件类型与消息触达要求。

### 主要风险
- 多平台数据口径不一致，影响统一展示与比较分析。
- 历史数据缺失导致异常检测初期准确率有限。
- 异常检测硬件部署节奏不确定，影响落地进度。

### 应对策略
- 先定义统一数据字典与映射规范。
- 用“规则可解释”优先于“模型复杂度”。
- 分阶段交付，先上线可用版本，再迭代优化精度。
