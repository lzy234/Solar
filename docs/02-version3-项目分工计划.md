# Version 3 项目分工计划（用于内部任务分配）

## 1. 项目目标（V3）

根据客户最新沟通结果，本阶段目标如下：

1. 打通多运维后台 API，统一采集电站数据。
2. 不做传统看板，核心信息直接推送到飞书。
3. 基于两组并联光伏板的电压/电流实时数据做异常检测。
4. 定制 OpenClaw：实现告警提醒、周报月报生成、工单建议能力。

## 2. 工作流与分工建议

| 工作流 | 负责人建议 | 主要任务 | 交付物 |
|---|---|---|---|
| 1. API 接入层 | 后端 A | 对接不同电站运维平台 API；统一数据模型；采集任务调度与重试 | `connector` 适配器、统一数据表结构、采集日志 |
| 2. 数据中台与存储 | 后端 B / 数据工程 | 实时数据入库；时序数据存储；告警事件表建设 | 数据库 Schema、ETL/同步任务、数据质量监控 |
| 3. 异常检测算法 | 算法工程师 | 设计“电流长期差异”检测规则/模型；阈值与时间窗口策略；误报控制 | 算法设计文档、离线评估报告、在线检测服务 |
| 4. 飞书推送能力 | 后端 C | 飞书机器人/应用接入；告警模板、日报/周报/月报模板；订阅策略 | 飞书消息服务、模板库、推送配置 |
| 5. OpenClaw Agent | AI 工程师 | 告警解读与提醒；自动生成周报月报；结合工单经验输出处理建议 | 3 个 Agent 流程、Prompt 与工具链、可观测日志 |
| 6. 工单与经验检索 | 后端 D + AI 工程师 | 新告警触发工单；历史案例检索；处理建议生成与反馈闭环 | 工单服务、知识库索引、建议命中率统计 |
| 7. 联调与验收 | QA + 全员 | 端到端场景测试；稳定性测试；灰度上线 | 测试用例、验收报告、上线清单 |

## 3. 里程碑计划（建议 6 周）

1. 第 1 周：需求冻结 + API 清单 + 数据字段标准确定。
2. 第 2 周：完成首批平台接入，数据入库流程跑通。
3. 第 3 周：异常检测 V1（规则版）上线，飞书告警推送打通。
4. 第 4 周：OpenClaw 周报/月报 Agent 上线，工单流程跑通。
5. 第 5 周：历史经验检索 + 告警建议 V1，误报率优化。
6. 第 6 周：联调、压测、客户演示版本发布。

## 4. 关键依赖与风险

1. 客户 API 文档完整性、联调环境稳定性、权限开通节奏可能影响整体排期。
2. 历史数据量不足会影响算法效果，建议先上线规则版再迭代模型版。
3. 飞书安全策略、消息频控策略需提前确认，避免上线后被限流。
4. 工单历史样本格式不统一时，需要先进行结构化清洗与标准化。

## 5. 本周可启动任务（建议）

1. 后端 A/B：完成“统一数据模型 + 1 个电站 API 样板接入”。
2. 算法工程师：输出“电流差异异常检测 V1 规则草案”。
3. AI 工程师：定义 3 个 Agent 的输入输出协议（告警/报告/工单建议）。
4. 后端 C：完成飞书机器人建联与消息模板 PoC。
5. PM/QA：整理验收场景，覆盖告警、报告、工单闭环三条主链路。

## 6. 会议沟通建议（可直接复用）

建议在内部同步会上按以下顺序沟通，以便快速达成分工共识：

1. 先确认本阶段目标边界：不做看板，以飞书触达为核心。
2. 再确认依赖项优先级：API 权限与字段标准先行。
3. 最后按工作流认领负责人，明确每周交付物与验收口径。
