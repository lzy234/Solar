## 问题现象

- 这次只聚焦“告警功能”接入，且项目处于初期，不追求把当前 demo 的全部丰富效果做实，只要先把告警主链路跑通即可。
- 当前前端里的告警页、告警详情、状态流转都还是本地模拟数据和本地状态，没有真实接口层。
- 如果继续按完整产品形态估算，会把 AI 对话、发电看板、实时推送、工单命中这些非 MVP 内容一起算进去，导致工期虚高，不适合当前阶段决策。

## 根因分析

- 后端文档里与“告警 MVP”直接相关的接口其实已经够用：
  - `GET /api/alerts`
  - `GET /api/alerts/{alert_id}`
  - `POST /api/alerts/{alert_id}/ack`
  - `POST /api/alerts/{alert_id}/close`
  - `GET /api/inverters/{sn}/trend`
- 当前前端告警功能的主要问题不是“后端能力不够”，而是前端还没有最基础的接入层：
  - 没有请求封装
  - 没有接口类型定义
  - 没有把后端字段转换成页面可用结构
  - 没有加载态、空态、错误态
- 文档里的 Redis Pub/Sub 不适合浏览器直接接入，但这对 MVP 不是阻塞问题，因为项目初期完全可以先不用实时推送，只做“进入页面拉取 + 手动刷新 + 简单轮询”。
- 当前 UI 中一部分告警详情字段没有后端来源，例如 AI 置信度、推荐动作、历史工单命中、预计损失、推送状态。这些内容在 MVP 阶段都可以先去掉或用静态占位，不应计入本阶段核心工作量。

## 修复方案

### MVP 目标边界

- 保留：
  - 告警列表
  - 告警详情基础信息
  - 告警趋势图
  - 告警关闭
  - 简单 ack
- 暂不处理：
  - Redis 实时推送
  - AI 对话联动
  - 工单命中
  - 首页发电总览
  - 复杂筛选器
  - 高保真状态编排

### 前端实际工作项

| 工作项 | 说明 | 预估 |
|---|---|---|
| 接口基础层 | 增加 `baseURL`、请求方法、统一错误处理 | 0.5 人天 |
| 告警数据类型与适配 | 把后端 `severity`、`status`、`detected_at` 等转换成前端可直接渲染的数据 | 0.5 人天 |
| 告警列表接入 | 用 `GET /api/alerts` 替换本地 seed，先支持默认列表和手动刷新 | 0.5 ~ 1 人天 |
| 告警详情接入 | 用 `GET /api/alerts/{id}` 和 `GET /api/inverters/{sn}/trend` 支撑详情弹层 | 0.5 ~ 1 人天 |
| 告警操作接入 | 详情页按钮接 `ack`、`close`，补 loading 与结果提示 | 0.5 人天 |
| 简单状态收口 | 把当前前端 `pending / processing / resolved` 先压缩映射到后端状态，避免复杂状态设计 | 0.5 人天 |
| 联调回归 | 校验列表、详情、趋势、关闭动作、异常响应 | 0.5 人天 |

### MVP 建议做法

1. 不要先拆很重的架构，够用即可。
2. 但至少要把 `App.tsx` 中告警相关数据抽到独立模块，避免接口代码直接糊进大文件。
3. 实时能力先不做，最多加一个 30 到 60 秒轮询。
4. 详情页中没有后端来源的高级卡片先删掉或隐藏，只保留真实数据能支撑的内容。
5. “处理中”这种中间态先不强做闭环，避免前后端状态模型不一致。

### 综合工期判断

- 如果只做“可演示、可联调、能查能关”的告警 MVP，前端工作量约 **2.5 ~ 4 人天**。
- 如果顺手补一点体验，例如简单筛选、轮询刷新、空态/错误态更完整，前端工作量约 **4 ~ 5 人天**。
- 这个估算已经默认：
  - 后端接口可直接访问
  - `alerts` 与 `trend` 返回结构稳定
  - 不额外新增鉴权、BFF、实时通道

## 影响范围

- 主要影响 `mobile-app/src/app/App.tsx` 的告警列表、告警详情和按钮动作。
- 当前 AI 对话页和发电看板可以暂时不动，不应成为本阶段阻塞项。
- 如果后续继续迭代，再从这版 MVP 上逐步补实时、补 AI、补高级分析即可。

## 验证结果

- 已按接入文档重新收敛了范围，只统计告警 MVP 必需的前端工作，不再把看板和 AI 功能计入本阶段。
- 已确认这份后端文档足以支撑一个“低完成度但可用”的告警功能前端版本。
- 已确认 Redis 实时订阅不是本阶段必需条件，可用手动刷新或简单轮询替代。

## 后续注意事项

- 真正开始开发前，最好先和后端确认 `GET /api/alerts/{id}` 与 `GET /api/inverters/{sn}/trend` 的返回字段样例，避免前端适配时反复返工。
- 如果产品希望保留现在 demo 里的丰富告警详情卡片，需要单独列为下一阶段需求，不应混入这次 MVP 工期。
- 如果后续需要浏览器端实时告警，再单独评估 SSE / WebSocket，不建议在当前阶段直接把 Redis 接入问题扩大化。
