## 问题现象

- 需要评估 `docs/光伏监测系统接入Agent指南.md` 这套后端接入到当前 `mobile-app` 原型后，前端到底要做哪些改造，以及哪些现有效果无法直接由这套后端支撑。
- 当前前端并不是“已接接口待联调”的状态，而是单文件演示原型：首页看板、告警详情和 AI 对话都由本地静态数据驱动，告警状态流转也是本地 `useState` 直接改值。
- 如果直接开始联调，不先拆清“可直接接的能力”和“需要补接口/补中间层的能力”，很容易出现页面做了一半才发现没有数据源，或者浏览器端根本无法按文档里的 Redis 方式接入。

## 根因分析

- 当前 `mobile-app/src/app/App.tsx` 内部直接声明了 `weeklyGenerationData`、`stationPowerData`、`alertsSeedData`、`initialMessages` 等演示数据，仓库内也没有 `fetch`、`axios`、`react-query`、`SWR`、`EventSource`、`WebSocket` 等真实数据接入代码，说明现在没有现成的数据层、接口层和实时订阅层。
- 当前告警处置逻辑是前端本地状态切换：`handleAlertStatus` 只做 `setAlerts`，并没有调用后端 `ack` / `close` 接口。这意味着“处理中”“已解决”这些 UI 状态与后端文档里的 `OPEN` / `RECOVERED` / `CLOSED` 不是同一套模型。
- 接入指南给出的实时能力是 Redis Pub/Sub，但浏览器前端不能直接安全地订阅 Redis。因此 H5/移动前端不能照文档原样接，只能改成：
  - 通过 HTTP 轮询刷新，或
  - 由后端/BFF 再转成 SSE / WebSocket 给前端。
- 当前 demo 中一批核心展示内容，文档里没有对应接口：
  - 发电总览/周趋势 `weeklyGenerationData`
  - 电站发电排行/容量完成率 `stationPowerData`
  - AI 诊断结论、置信度、推荐动作、历史工单命中、预计损失、推送状态
  - 多轮 AI 对话本身
  这意味着“按现有 UI 原样接后端”并不可行，前端必须降级展示，或者反向推动后端补接口。

## 修复方案

### 可直接落地的前端工作

| 工作项 | 说明 | 预估 |
|---|---|---|
| 建立接口层和环境配置 | 增加 `baseURL`、请求封装、错误处理、时间格式化、后端枚举到前端枚举的转换 | 0.5 ~ 1 人天 |
| 抽离类型与数据适配层 | 为 `alerts`、`alert detail`、`station status`、`inverter latest`、`trend` 建立 TypeScript 类型和 adapter | 0.5 ~ 1 人天 |
| 告警列表接入 | 把现有 `alertsSeedData` 改成接口数据，支持筛选、分页/增量加载、空态和错误态 | 1 人天 |
| 告警详情接入 | 对接 `GET /api/alerts/{id}` 与 `GET /api/inverters/{sn}/trend`，重做详情字段映射 | 1 人天 |
| 告警处置动作接入 | 把 `ack`、`close` 接到按钮行为，补提交中、成功、失败反馈 | 0.5 ~ 1 人天 |
| 电站状态接入 | 将顶部总览和局部统计改为 `GET /api/stations/{station_id}/status` 可提供的字段 | 0.5 人天 |
| 刷新策略 | 设计手动刷新、页面进入刷新、定时轮询策略 | 0.5 人天 |
| 联调与回归 | 校验接口异常、状态切换、图表时间范围、中文文案 | 0.5 ~ 1 人天 |

### 接口不足导致的额外工作

| 缺口 | 对前端的影响 | 前端处理方式 |
|---|---|---|
| Redis 只能给服务端订阅 | 浏览器不能直接实时收告警 | 先做轮询，或等待后端提供 SSE / WebSocket / 网关转发 |
| 没有“处理中”状态 | 现有 UI 的 `pending / processing / resolved` 不能直接复用 | 重定义状态文案，或把“处理中”做成纯前端临时态 |
| 没有发电周趋势/排行接口 | 首页发电看板无法真实化 | 删除、降级成占位，或推动后端补接口 |
| 没有 AI/工单知识库接口 | AI 对话页、原因分析、工单命中只能继续 mock | 保留演示模式，或新增独立 AI/BFF 服务 |
| 电站状态接口字段较少 | 很多 KPI 卡片没有数据来源 | 收缩看板内容，只保留在线设备数、开放告警数、最新告警 |

### 推荐的实施顺序

1. 先把 `App.tsx` 里的静态数据层拆出来，至少拆成 `types`、`api`、`adapters`、`hooks` 四层，否则一边联调一边继续堆单文件，后面会很难维护。
2. 第一阶段只做 REST MVP：告警列表、告警详情、趋势图、站点状态、关闭告警。
3. 第二阶段再决定实时方案：如果后端短期不补 SSE / WebSocket，就前端先轮询。
4. 第三阶段单独评估 AI 页和高级看板，是否继续保留演示数据，还是要求新后端/新服务支持。

### 综合工期判断

- 如果目标是“把当前 demo 的告警主链路接上真实后端，允许部分页面降级”，前端工作量约 **4 ~ 6 人天**。
- 如果目标是“尽量保留现在 demo 的丰富效果，并做真实实时推送、AI 分析、工单命中、完整看板”，仅前端侧约 **7 ~ 10 人天**，且明显依赖后端新增接口或新增中间层服务。
- 上述评估只包含前端，不包含后端补 SSE / WebSocket、AI 服务、知识库服务、工单系统打通的开发量。

## 影响范围

- `mobile-app/src/app/App.tsx` 大概率不适合继续单文件联调，至少需要拆出接口访问、类型定义、数据转换和页面状态管理。
- 首页看板、告警详情页、AI 对话页都会受影响，其中真正能直接接这份后端文档的主要是“告警相关”和“趋势图相关”部分。
- `codingLog` 新增了 `docs/mobile-app` 分类，用于沉淀这类分析性记录，而不是混进 `feature` 或 `bugfix`。

## 验证结果

- 已核对 `docs/光伏监测系统接入Agent指南.md` 中可用接口：`alerts`、`alert detail`、`ack`、`close`、`inverter latest`、`trend`、`station status`。
- 已核对当前 `mobile-app` 代码现状：仓库内没有真实接口调用或实时订阅代码，`App.tsx` 以本地 seed 数据和本地状态流转为主。
- 已确认浏览器前端不能直接按文档方式消费 Redis Pub/Sub，这部分必须改接入方案。

## 后续注意事项

- 开始联调前，最好先和后端确认是否愿意补一个前端可消费的实时通道（SSE / WebSocket），否则只能接受轮询方案。
- 需要尽早做一次“页面保留清单”确认：哪些演示卡片可以删，哪些必须保留，避免前端为了无接口的数据结构空转。
- 如果 AI 对话页要从 demo 变成真实能力，应该单列一个服务边界说明，因为这份光伏监测后端文档本身并没有提供聊天或知识库检索接口。
